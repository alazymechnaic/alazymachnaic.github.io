Towards Automatic and Precise Heap Layout Manipulation for General-Purpose Programs

论文链接：https://www.ndss-symposium.org/wp-content/uploads/2023/02/ndss2023_s232_paper.pdf

摘要

​ 文章实现了一个通用的堆布局操控工具BAGUA，该工具将堆问题进行建模，将堆操作建模为线性规划问题，从而确定堆原语的调用序列，最终实现目标程序的堆布局。同时BAGUA能够求解实现相应布局的程序输入。

漏洞原理

​ 上述代码中在44行缺少对size大小的检查，导致的堆溢出漏洞。在42行选择最小的size复制给toread，在行按照toread进行内存的读入。在ff_rtmp_packet_create()函数中进行内存块的申请和创建（这里realloc函数的第一个参数如果是NULL，则会执行malloc的操作，分配第二个参数大小的堆块），并且在ffurl_read_complete函数中进行溢出。

​ （原文中并没有对该漏洞原理的详细展示，这里参考网上对于CVE-2016-10191 漏洞的分析过程https://blog.csdn.net/weixin_33795743/article/details/88979345）。在RTMP协议中发送数据包的最小单位是一个chunk。客户端和服务器会互相协商好发送给对方的 chunk 的最大大小，初始为 0x80 个字节。一个 RTMP Message 如果超出了Max chunk size, 就需要被拆分成多个 chunk 来发送。在 chunk 的 header 中会带有 Chunk Stream ID 字段（后面简称 CSID），用于对等端在收到 chunk 的时候重新组装成一个 Message，相同的CSID 的 chunk 是属于同一个 Message 的，后续会在堆中使用realloc进行再次分配一个更大的chunk。在每一个 Chunk 的 Message Header 部分都会有一个 Size 字段存储该 chunk 所属的 Message 的大小，按道理如果是同一个 Message 的 chunk 的话，那么 size 字段都应该是相同的。这次漏洞的起因是对于属于同一个 Message 的 Chunk的 size 字段没有校验前后是否一致，导致写入堆的时候缓冲区溢出。

typedefstructRTMPPacket {
intchannel_id; ///< RTMP channel ID (nothing to do with audio/video     channels though)

uint32_t       ts_field;   ///< 24-bit timestamp or increment to the    previous one, in milliseconds (latter only for media packets). Clipped to a  maximum of 0xFFFFFF, indicating an extended timestamp field.
uint32_t       extra;      ///< probably an additional channel ID used  during streaming data    //这个是 Message Stream ID？
uint8_t        *data;      ///< packet payload
int            size;       ///< packet payload size
int            offset;     ///< amount of data read so far
int            read;       ///< amount read, including headers
} RTMPPacket;

​ 因此利用思路就是程序中会使用RTMPacket结构体存储当前的一个chunk，程序在接收到数据后会把内容存放在data buffer上，如果能够溢出buffer，一直写到下面的data指针，那么data指针就可以被控制，后续在向data指针写入内容时就可以实现任意地址写。因此这里要buffer要在chunk数据结构的上方，才能够让其溢出覆盖。

​ 程序中存在两个逻辑原语：

# 当接收到一个ID的数据时，申请堆内存创建相应的chunk记录结构
p1: A1(∀)—A2(∀)
# 申请一个data buffer进行存储原始数据，但是这里使用的是realloc函数，因此呈现出free ， malloc 后 free的执行顺序
p2: A1(∀)—F(p1:A1)—A2(∀)

可以看到由于程序逻辑，导致p1只会执行一次，而p2可以执行多次（这是由于程序本身的特性所决定的）。程序中需要参考上述两个原语构造出合适的堆布局，需要buffer在RTMPPacket结构体之上。

文章要解决的问题

如何精准识别程序中的堆操作原语？

  • 解决方法：利用Heap Operation Dependence Graph来识别原语。首先使用以堆原语引导的模糊测试方法来查找堆操作原语，同时利用这些原语来构建HODG，其中的每一个结点都是一个堆操作原语，边是控制流和数据流。还可以通过该方法识别dispatcher。

如何操控有限的堆原语来进行内存块的挖掘和填充？

  • 除了堆操作原语代码本身的限制，堆操作原语之间在逻辑上可能存在相互依赖的关系，限制了堆原语的布局操控能力。我们自然是希望堆原语中就只存在需要的单一堆操作就好，但堆操作往往是相互关联的。
  • 解决方法：对堆管理器进行建模

如何对抗堆原语在执行后造成的副作用？

  • 由于堆管理是根据实时的堆布局状态所决定的（例如当前已分配的内存，空闲内存块链表等）。这类特性很难对对管理器的运行方式进行建模。
  • 解决方法：通过对堆分配机制进行建模，添加新的约束来控制堆布局

BAGUA设计

  • 输入：目标程序，能够造成程序crash的输入，期望的堆布局
  • 输出：原语的调用序列

解决问题：在生成目标程序的序列时，应该知道选取哪些原语，以及这些原语应该按照什么标准进行怎样的排序。

  • 需要哪些原语？这些原语要被使用多少次？
    • 需要了解原语触发的约束以及目标布局的约束。原语的触发次数t也必须要满足程序本身的执行逻辑约束，目标布局必须满足“利用d内存空缺来填补占据其他的内存块”的约束。
    • 构建线性方程组，设定目标函数进行约束求解！！！

原语识别模块

  • 构造HODG：
    • HODG之中包含了程序的数据流以及控制流的部分，每一个结点表示一个堆操作，每一个边表示控制流。（这不就是把CFG给简化了么，只留下了堆操作的代码？）
    • 下图中能够看到，目标程序中存在的anchor node其实就是类似调度器一般的结点。图中存在三个原语，每一个原语的entry node表示当前原语的进入结点，该图表示，E1，N1，N2为一个原语，并且由N2指向E1的边来表示原语的结束。并且P3 的执行要以P1的执行为前提。因此P1与P3能够按照先后顺序组成一个执行序列。但是P1与P2就不能组成一个序列。
  • 面向堆操作的FUZZ引擎：
    • FUZZ倾向于在目标程序中尽可能多地丰富当前的heap operation set
    • After that, BAGUA determines which pointer is freed on free() using the execution traces and uses symbolic execution to recover the size of allocation on malloc(), calloc() and realloc().

      • 例如p1与p2只能按照前后顺序进行执行。

能力建模

  • State-Varied Feature of Heap Allocators

    • 堆操作本身的复杂特性，导致当前堆管理器对于一个堆块的操作取决于当前堆布局的状态（例如当前的空闲链表等
    • 本文将其定义为 state-varied feature ，例如在某些情况下申请size A的内存，加入空闲链中存在size A的chunk，那么就直接分配，如果空闲链中不存在size A的chunk，那么就会去找更大的chunk 链中的chunk进行分配。
    • 这里文中配了一个例子：
    • 在不同的堆状态下申请一块0x200大小的内存产生不同的效果。
      • a：bin中恰好存在一个0x200大小的chunk，则直接拿下
      • b：bin中不存在0x200的chunk，但是存在0x400的chunk，就把0x400的chunk拆分并产生一个0x200的hole（文章中称其为噪音noise）
    • 因此在state-varied feature的作用下，程序中的堆分配变得更复杂了，同时也为堆操作提供了些额外的能力（例如，malloc操作本身应该是填洞的，但是在b状态下变成了填洞和挖洞两种功能，本文称其为extending capability

    Basic Capability

    • 这里的c表示的是一条空闲chunk链，其中的A表示allocate，f表示free操作。x表示当前链的size，y表示要alloc/free的size。也就是如果当前的chunk size相同就会对当前chain +1/-1操作，如果不是则操作为0.这里仅仅作为堆操作原语的基础能力，extending capability由当前堆状态所决定

    Extending Capability

    • 其实这里作者将扩展能力定义为基础能力的副作用（side effect）,这里对于原语能力的利用可以理解为，在申请size为x的操作时，去改变size 为y的链上的情况。能够看到Allocate不一定是填洞，Free也未必是挖洞。

    • Allocate

      • 这里的+1和-1分别表示在目标链上进行创建空闲块和拿掉空闲块的操作。很容易理解的情况
        • 如果想要在Cy上增加一个chunk，那么只有当Cx上chunk为空，且y = z - x 的时候，也就是要申请的size没有空闲chunk了，从更大的chunk中切割出来，剩余了一个大小为y的chunk，那么就将其放入到Cy上，导致了Cy + 1。
        • 如果想要在Cy上减少一个chunk，要么是x = y，直接分配一个y大小的chunk。要么就是分配x时，Cx为空，从Cy上取出一个chunk进行切割。

    • Free

      • 通过free实现的几种情况也十分好解释
        • +1: free时，只有当目标chunk的size是y的时候，才能让Cy+1。另一种情况时z = y，也就是别x触发合并操作导致合并到size y上的时候。
        • -1：如果当前free的chunk和Cy中的chunk是相邻的，那么会触发chunk合并，合并为size z的chunk，Cy -1。
        • -2：如果free掉的chunk两边都是size y的chunk，那就引发两次合并，导致Cy - 2。
