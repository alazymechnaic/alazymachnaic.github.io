Go语言编程

基础语法

go语言本身是编译型语言，具备高效、简洁、并发能力强的特点，因此常常被用于云计算、网络服务。Go语言在执行的过程中，将Go代码转化为中间表示，并将程序的中间表示转化为目标平台的机器码。

    fmt.Printf("Congratulations! Go has successfully been installed on your system\n")

运行代码

 go run hello.go

拆分过程后进行代码的运行

 go build -x hello.go  # 拆分过程实现，会产生一个名称为hello的二进制文件

通过gdb调试二进制文件能够看到go产生的中间语言。go语言在运行的过程中通过go的runtime与内核进行通信，runtime完成包括内存的分配等操作。

Go程序首次进入的函数是runtime.rt0_amd64，通过runtime.rt0_go函数调用main函数，并完成数据变量的初始化，CPU核心数量的获取以及全局变量m0、g0的设置。而后m0开启第一个线程。Go语言的runtime中存在优秀的线程管理机制，

  • G：goroutine，⼀个计算任务。由需要执⾏的代码和其上下⽂组成，上下⽂包括：当前代码位置，栈顶、栈底地址，状态等。（协程）
  • M：machine，系统线程，执⾏实体，想要在 CPU 上执⾏代码，必须有线程，与 C 语⾔中的线程相同，通过系统调⽤ clone 来创建。
  • P：processor，虚拟处理器，M 必须获得 P 才能执⾏代码，否则必须陷⼊休眠(后台监控线程除外)，你也可以将其理解为⼀种 token，有这个 token，才有在物理 CPU 核⼼上执⾏的权⼒。

在go语言中的指针操作与C语言类似，可以通过:=符号完成对变量的简单声明一个指针类型的变量。

    *v = 40

在上述过程中能够看到，Go语言只能对程序中的int*类型的变量做解引用操作，且与C语言不同，指针本身并不能像数值一样直接运算。

由于Go语言的特性，直接读取给定地址的内容是不允许的，因为Go语言具有内存安全性和类型安全性的限制。Go语言的内存访问是受限制的，不能直接操纵指针或进行低级别的内存操作。直接将数字转化为指针是不安全的，因此编译都不会通过。

Tunter: Assessing Exploitability of Vulnerabilities with Taint-Guided Exploitable States Exploration

主要贡献点

  • 基于污点分析的候选exploit状态探索策略
  • 用户缓解搜索状态爆照的剪枝策略
  • 在14个CTF上，2个real world上测试，在Revery上的测试比其更快但是并没有找全面

关键思想

  • 一旦发生内存错误就会导致程序变为一个奇怪的状态机，发现exploit的本质是搜索这个状态机中的空间，但是搜索这该状态空间是十分耗时的
  • 文章指出的问题
    • 使用符号执行进行求解只能对小型的程序起作用，对于大一点的程序会直接路径爆炸（但是这个我们解决了，使用启发式的路径探索策略，有目标的进行探索和评估
    • 对于fuzz的使用，Revery使用fuzz的方式进行可利用状态的探索，太耗时
  • 使用启发式的状态剪枝方法来防止状态爆炸，构造目标可利用状态的约束

相似之处

  • 都是使用污点分析标记受污染的内存，在传播的过程中找到受影响的sink（例如，memcpy参数，ret地址等），作为候选的状态
  • 使用动静转换策略，找到程序漏洞的触发位置，将动态执行的状态转换为静态执行的状态，从静态位置开始进行污点分析，这一部分与我们的方法一致，但是并没有完全的dump内存，而是模拟了一个相似的静态程序状态
  • 数据流分析时的指针问题，因为rax可控，所以直接将rsi定义为可控，在下面的指令中mov [rsi],rbx就认为是个gadget，我们也是这么做的。。。，但是不同之处在于后续进行求解的过程中会在数据流分析中施加内存搜索策略

主要的问题

  • 所有的案例都是在CTF上进行实现，两个真实世界的案例，构造了一个Revery无法进行实现的案例的exp，且多数攻击都是通过改变控制流实现的，

  • 行文中更多的是像对Revery研究的补充，而并非是一篇独立的文章，并且在比较结果上并没有比Revery好多少

  • 文章对漏洞的理解，将内存错误视为对程序原有状态机的改变，我们认为内存错误的本质是对合法数据流依赖关系的破坏

  • 文章中通过控制流劫持的方法进行攻击，我们使用的是满足控制流完整性的方式进行攻击，限制条件更多，后面的实验最好能找到原本没有，算法构造出的

  • 污点分析找路径时的爆炸问题并没有进行考虑，而我们使用启发式的路径排序策略，缓解了这一问题（他是用修剪不相关路径实现的？

  • 符号执行的路径修剪策略潦草：其实就是在指定了路径后去求解，出现循环太多次的就剪掉，这一部分工程实现讲得有点多

  • 调研量过少，涉及到的AEG部分内容理解不完善

  • 我们的优势在于保证所有保护开启的情况下完成的攻击，该文章则是在保护全关的情况下进行利用

  • 对于精准的内存地址，没有办法做到求解，page 9

  • angr的求解问题，并不能做到精准的求解

Hook so加载（dlopen）

// 当遇到“等so加载完后进行某Hook操作时”，可以Hook dlopen（有的是android_dlopen_ext）方法来确定so的加载。PS：不推荐Hook System.loadLibrary()，因为加载so的方式并不止这一种，而且最终都会调用 dlopen()
function hookDlopen() {
    let dlopenAddr = Module.findExportByName(null, "dlopen");
    if (dlopenAddr) {
        Interceptor.attach(dlopenAddr, {
            onEnter(args) { // dlopen(const char* filename, int flags)
                let soName = args[0].readCString(); // "/data/app/com.zyc.fridasodemo-1/lib/x86/libnative-lib.so"
                if (soName.indexOf("libnative-lib.so") != -1) {
                    console.log("成功加载到了-->" + soName);
                    this.hasloaded = true;
                }
            },
            onLeave(retval) { //onLeave()中才是dlopen()加载完成后
                if (this.hasloaded) {
                    hooImEx(); // 等so加载完成就执行导入/导出函数打印
                }
            }
        });
    }
}

function hookByName() {

            Interceptor.attach(address, {

​ 但是frida一直是NOT FOUND，这是因为frida要重启APP去执行该脚本，在onCreate函数执行前libnative-lib.so还没有加载，因此在程序内存中是找不到该so文件的。因此有两种方法，一种是使用dlopen手动加载该so，但是程序直接崩溃了。。。。

​ 另一种方法是hook onCreate函数，在onCreate函数执行之前执行native的hook，当然这里so文件已经加载进内存了，stringFromJNI中的字符串是硬编码存储，也可以直接修改内存。要注意以下几点：

  • hook native
    • 在对native函数进行hook的时候要使用函数的全名。打开IDA后Export显示的名称
    • 修改返回值时，如果返回值是string，不能简单的replace（具体方式看下面脚本）
  • 修改内存
    • 修改内存先找so基址，偏移量将IDA segment中的base addr设为0后，显示的地址加so基址即可
    • 写内容时先修改内存页的保护属性

Hook重载函数

// 由于Calc中有add(int num1)和add(int num1,String num2)，像普通函数那样Hook是会报错的，涉及到重载就要用到 overload() ：
// 这里按照传入的参数类型区分二者的差异
Java.perform(function(){
    var clazz = Java.use("com.zyc.fridademo.Calc");
    clazz.add.overload("int").implementation = function()
    {
        console.log("Hook add(int num1)...");
        return 888;
    }

    clazz.add.overload("int","java.lang.String").implementation = function()
    {
        console.log("Hook add(int num1,String num2)...");
        return 999;
    }
});
// 也可以通过遍历重载函数的形似	
Java.perform(function(){
    var clazz = Java.use("com.zyc.fridademo.Calc");
    var count = clazz.add.overloads.length;
    for(var i=0;i<count;i++){
        clazz.add.overloads[i].implementation = function(){
            arguments[0] = 8;
            if(arguments[1]){
                send(arguments[1]);
            }
            return this.add.apply(this,arguments);
        }
    }
});

重写Tosat

​ 参考这个例子能看到，js可以完全重写Android java层的代码。

Java.perform(function () {
  // hook类: MainActivity
  var MainActivity = Java.use("com.yashu.aaaaaaaaaaaaaaaaaaaaaa.MainActivity");
  // hook方法: MainActivity.onCreate
  MainActivity.onCreate.implementation = function (savedInstanceState) {
    console.log("enter onCreate");
    // 不打断程序, 该执行的还要执行
    // 执行this.onCreate, 且传参
    this.onCreate(savedInstanceState);
  };

  // 获得Toast组件
  var Toast = Java.use("android.widget.Toast");
  var makeText = Toast.makeText;
  var String = Java.use("java.lang.String");
  // 函数重载, 设置参数类型
  makeText.overload("android.content.Context", "java.lang.CharSequence", "int").implementation = function (
    context,
    content,
    time
  ) {
    console.log("修改toast内容");
    // 设置字符串重复次数
    var content = "牙叔教程 超级棒棒\n".repeat(10);
    // 实例化字符串
    var hookContent = String.$new(content);
    // 可以hook, 但是不能打断原先的程序, 原来该做什么, 还要继续做下去
    return this.makeText(context, hookContent, time);
  };
});
